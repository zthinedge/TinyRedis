# 架构决策记录

本文件记录关键决策及其原因。

## ADR-001：按分层推进（`core -> protocol -> command -> server`）

状态：Accepted

原因：

- 每层都能独立测试，定位问题更快。
- 符合“逐步理解 Redis 各模块”的学习路径。
- 更适合 AI 协作：任务边界清晰、单次改动可控。

## ADR-002：先只做 RESP2

状态：Accepted

原因：

- RESP2 已足够覆盖当前核心学习目标。
- 有助于控制范围，先跑通端到端主链路。
- 等命令管线稳定后再扩展 RESP3。

## ADR-003：优先单线程

状态：Accepted

原因：

- 学 Redis 设计，先从单线程事件循环开始最直观。
- 基础能力未稳定前可避免竞态问题。
- 调试更简单，测试结果更具确定性。

## ADR-004：每个功能都必须配套测试

状态：Accepted

原因：

- 在 AI 高速迭代时防止回归。
- 提升重构安全性（尤其 parser 和 dict 内部逻辑）。
- 复杂度逐步上升时保持信心和节奏。
